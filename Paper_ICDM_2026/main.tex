\title{CryptoRegimeShift-LOB: A Regime-Aware L2 Crypto Benchmark for Forecast-to-Execution Degradation}

\author{\IEEEauthorblockN{Anonymous Authors}}

\maketitle

\begin{abstract}
Limit order book (LOB) forecasting models are often evaluated using directional metrics, yet such metrics can overstate deployable signal when spreads, fees, latency, visible-depth constraints, and regime shifts erode short-horizon edge. We present CryptoRegimeShift-LOB, a snapshot-level L2 crypto order book benchmark and evaluation protocol for studying forecast-to-execution degradation under microstructure regime shifts. The benchmark covers full-year BTC-USDT and ETH-USDT L2 snapshots and connects causal feature construction, cost-aware ternary labels, regime-aware forecasting, L2 snapshot replay, stress testing, bootstrap diagnostics, and BTC$\leftrightarrow$ETH cross-asset validation. We include Robust Selective Execution Policy (RSEP) as an interpretable selective-execution diagnostic, not as a universal trading policy. The empirical evidence shows that forecasting signal can be visible while net PnL remains negative after execution frictions; RSEP mitigates losses in some evaluated settings but does not demonstrate profitability. The central finding is methodological: LOB benchmarks for financial decision contexts should be cost-aware, regime-aware, stress-tested, and explicit about the observational limits of L2 data.
\end{abstract}

\section{Introduction}
\IEEEPARstart{L}{imit} order book (LOB) forecasting is a core data mining task in high-frequency markets because the book records visible supply, demand, and liquidity at fine temporal resolution. Standard evaluation trains classifiers for future mid-price movement and reports accuracy, macro-F1, Matthews correlation coefficient (MCC), or balanced accuracy. These metrics measure predictive structure, but they do not establish whether a signal remains actionable after spread crossing, explicit fees, latency, limited visible depth, and regime-dependent liquidity. A forecast can be directionally correct while still failing along the path from prediction to execution.

This gap is microstructural rather than purely predictive. Trading outcomes depend on trading mechanisms, liquidity provision, adverse selection, order-book depth, and the speed at which market participants react to information \cite{madhavan2000microstructure,ohara2015highfrequency,gould2013limitorderbooks}. In high-frequency settings, small timing changes or spread changes can alter the realized value of a short-horizon signal \cite{budish2015hftarmsrace,aquilina2022armsrace,bis_hft}. Crypto markets make this problem salient because trading is continuous, liquidity conditions can change sharply, and market-integrity concerns remain central \cite{almeida2024cryptomicrostructure,iosco_crypto2023}. A benchmark for LOB models should therefore ask not only whether a model predicts movement, but also how that signal degrades under execution frictions and changing microstructure regimes.

Recent LOB forecasting studies have substantially improved short-horizon prediction. DeepLOB and related neural approaches model spatial and temporal structure in book states \cite{deeplob2019,sirignano2019deeplob,zhang2021marketbyorder}, while benchmark and comparative studies improve experimental discipline for mid-price prediction \cite{ntakaris2018benchmark,prata2024lobbenchmark,briola2025lobframe}. However, prediction-centered evaluation leaves a downstream question underexamined: when forecasts are exposed to transaction costs, latency shifts, spread widening, depth shocks, and regime changes, how much of the apparent predictive edge survives?

CryptoRegimeShift-LOB addresses this question as an applied benchmark for tracing short-horizon LOB forecasts through observable execution frictions. The data are snapshot-level L2 observations rather than event-level L3 or market-by-order records, so the execution layer uses a visible-depth market-order replay approximation. This boundary keeps the benchmark focused on reproducible degradation analysis while reserving hidden liquidity, queue position, passive fills, and live venue behavior for future data settings.

The paper makes four applied benchmark contributions. First, it provides a full-year BTC-USDT and ETH-USDT L2 snapshot benchmark with chronological splits, causal features, and locked dataset statistics. Second, it turns short-horizon LOB prediction into a cost-aware evaluation problem through ternary labels, interpretable microstructure regimes, and by-regime diagnostics. Third, it provides a reproducible forecast-to-execution path: fixed predictions are passed through L2 snapshot replay, validation-tuned execution gates, fee/latency/spread/depth stress, and day-level bootstrap. Fourth, it uses RSEP and BTC$\leftrightarrow$ETH transfer as diagnostic probes for when forecast signal survives, degrades, or fails under execution assumptions. The resulting benchmark is useful because it exposes failure modes that prediction-only LOB benchmarks hide.


This framing makes negative evidence an applied contribution. In financial machine learning, a model can appear attractive when evaluation stops at directional prediction, especially when a metric rewards frequent FLAT predictions or when minority UP/DOWN classes are hard to classify. The benchmark therefore separates three questions: whether a model captures predictive structure, whether the predicted movement is large enough to clear local costs, and whether the resulting action survives execution stress. A result with positive gross directional movement but negative net PnL localizes forecast-to-execution degradation rather than merely lowering a leaderboard score.

The cross-asset experiments add a second applied diagnostic. Reporting BTC$\leftrightarrow$ETH transfer by direction tests whether a source-tuned pipeline produces interpretable degradation on a related but different asset. The goal is not to average away asymmetry, but to expose how distribution shift, label balance, regime mix, and calibration affect the same evaluation path.

\section{Related Work}
\subsection{LOB Forecasting and Benchmarks}
LOB forecasting has been studied as short-horizon mid-price movement prediction. The FI-2010 benchmark helped standardize supervised LOB prediction \cite{ntakaris2018benchmark}; DeepLOB showed that convolutional and recurrent layers can extract order-book structure \cite{deeplob2019}; and later work extended the modeling space to deep LOB learning, market-by-order data, and transformer-style architectures \cite{sirignano2019deeplob,zhang2021marketbyorder,xiao2025lit}. Recent benchmark and microstructural guide papers emphasize rigorous comparison and feature interpretation \cite{prata2024lobbenchmark,briola2025lobframe}. These works motivate our forecasting baselines, but our novelty is not a new predictor. We use forecasting models as controlled inputs to an execution-aware benchmark.


A recurring lesson from this literature is that LOB models are sensitive to task definition. Horizon length, label threshold, class balance, and the number of visible levels can change both absolute performance and model ranking. This is why CryptoRegimeShift-LOB does not treat any single forecasting table as final evidence. Forecasting models are necessary to produce candidate signals, but the benchmark asks whether the same signals are robust after the cost-aware label construction, regime split, and execution replay are applied. In this sense, our use of DeepLOB-style literature is contextual: it motivates the input structure and model family, while the contribution of this paper is the evaluation path from prediction to degraded execution.

\subsection{Market Microstructure and Execution Frictions}
Market microstructure research shows that observed prices and realized trading outcomes depend on trading rules, visible and hidden liquidity, order flow, and adverse selection \cite{madhavan2000microstructure,ohara2015highfrequency,cont2010stochasticlob,cartea2015algorithmic}. HFT research further shows that latency and speed are economically meaningful frictions rather than implementation details \cite{budish2015hftarmsrace,aquilina2022armsrace,aquilina2025speedpremium}. For LOB forecasting, this implies that directional correctness is not enough: a signal must survive spread crossing, fee, depth, and timing risk. Because our data are L2 snapshots, we evaluate only a visible-depth market-order approximation and explicitly avoid L3 or live-execution claims.


Crypto L2 data intensifies this distinction. Snapshot data records visible price levels and sizes, which is sufficient to construct spread, depth, imbalance, and market-order sweep approximations. It does not record the complete order lifecycle. Therefore, the benchmark is deliberately conservative: it evaluates costs observable from visible depth and stress perturbations, while treating unobserved queue priority, cancellation races, and hidden liquidity as outside scope. This makes the empirical claims narrower but more defensible.

\subsection{Regime-Aware and Stress Evaluation}
Aggregate test metrics can hide failures under distribution shift. Robustness and dataset-shift studies show that fixed test performance can degrade under shifted conditions \cite{quinonero2009datasetshift,ovadia2019uncertainty,koh2021wilds}. Time-series evaluation also requires chronological validation to avoid leakage and overoptimistic estimates \cite{bergmeir2018timeseriescv,cerqueira2020timeserieseval}. Financial evaluation adds cost and data-snooping risks, motivating bootstrap and conservative backtest interpretation \cite{efron1979bootstrap,politis1994stationarybootstrap,white2000realitycheck,bailey2016backtestoverfitting,lopezdeprado2018afml}. CryptoRegimeShift-LOB combines these ideas in a domain-specific path: causal L2 features $\rightarrow$ cost-aware labels $\rightarrow$ regime-aware forecasting $\rightarrow$ execution replay $\rightarrow$ stress testing and cross-asset diagnostics.

\section{Data and Benchmark Construction}
\subsection{L2 Snapshot Data}
The benchmark uses Binance BTC-USDT and ETH-USDT L2 order book snapshots for 2024, following the Crypto Lake data documentation for historical crypto market data \cite{cryptolake_data}. Each row is a snapshot with 20 bid levels and 20 ask levels, represented by price and size fields, along with timestamp, sequence, exchange, and symbol metadata. The audit-level dataset contains 167,753,156 valid BTC-USDT snapshots and 114,416,283 valid ETH-USDT snapshots over 366 calendar days. After dropping rows that lack the required future horizon for labels, the feature/label stores contain 167,751,306 BTC-USDT rows and 114,414,433 ETH-USDT rows.

\subsection{Audit, Causal Features, and Labels}
Before feature construction, snapshots are audited for schema validity, timestamp validity, positive price and size fields, and non-crossed best quotes. These checks create a data version that can be linked to later forecasting, execution, stress, and bootstrap artifacts. The audit does not reconstruct missing message-level events; rows with invalid book states are cleaned or logged under the benchmark configuration.


The audit layer records interval statistics, spread statistics, depth statistics, duplicate timestamp counts, invalid price or size counts, and crossed-book counts. These values are not included merely as data-engineering metadata. They are part of the benchmark evidence path: when a result table reports forecast-to-execution degradation, the reader can trace it back to the audited snapshot scope and to the split used for model training and policy tuning. This is important for Applied Track evaluation because the contribution depends on repeatable data construction rather than a single isolated model run.

For each decision time $t$, features use only information available at or before $t$. Let $b^{p}_{t,\ell},b^{q}_{t,\ell}$ be bid price and size at level $\ell$, and let $a^{p}_{t,\ell},a^{q}_{t,\ell}$ be ask price and size. The mid-price and spread are
\begin{equation}
m_t=\frac{a^{p}_{t,0}+b^{p}_{t,0}}{2},\qquad s_t=a^{p}_{t,0}-b^{p}_{t,0}.
\end{equation}
The feature store contains spread, relative spread, multi-level depth, imbalance, order-flow-imbalance proxies, short-horizon returns and volatility, momentum, choppiness, liquidity-stress scores, and adverse-selection proxies. Rolling statistics end at $t$; feature scalers and model parameters are fitted on training rows, regime quantiles are fitted on the training prefix, policy thresholds are tuned on validation rows, and test rows are not used for fitting or selection.

Labels are cost-aware ternary targets. For horizon $h$,
\begin{equation}
r_{t,h}=\frac{m_{t+h}-m_t}{m_t+\epsilon},\qquad
 y_t=\begin{cases}
\mathrm{UP}, & r_{t,h}>\tau_t,\\
\mathrm{DOWN}, & r_{t,h}< -\tau_t,\\
\mathrm{FLAT}, & \mathrm{otherwise},
\end{cases}
\end{equation}
where $\epsilon=10^{-9}$. The benchmark uses a fixed local threshold
\begin{equation}
\tau_t=(1+\kappa)\rho_t+\frac{f}{10000},\qquad
\rho_t=\frac{s_t}{m_t+\epsilon}.
\label{eq:label_threshold}
\end{equation}
The default benchmark constants are $h=50$ events, $f=1.0$ bps, and $\kappa=0.5$, so $\tau_t=1.5\rho_t+0.0001$. The same constants are used for BTC-USDT and ETH-USDT. They are fixed in the benchmark configuration before any split is evaluated; they are not fitted, validation-tuned, or adjusted using test data. The inequalities are strict: equality with either boundary is assigned to FLAT.

This design makes the FLAT class meaningful: it is not a residual class for uncertain predictions, but a statement that the future move is too small to clear the fixed local cost proxy. Therefore, UP and DOWN labels represent movements that are directionally and cost-aware significant under the chosen horizon. Trade size, latency, and visible-depth sweep are deliberately excluded from $\tau_t$; they enter only later in the execution simulator.

The future return is used only to define the supervised target, not as a feature, regime input, normalization input, or policy decision variable. The benchmark uses chronological splits: training fits model parameters, feature scalers, and regime quantiles; validation tunes model-selection thresholds, execution thresholds, and RSEP parameters; and test is reserved for final reporting. The locked Stage 3 artifacts are row-contiguous chronological splits and the split audit documents a 50-event label-horizon overlap at train--validation and validation--test boundaries. To make subsequent reruns strictly purged, the split generator now drops the last $h$ rows before each later split boundary.

The taxonomy audit reports \texttt{UNKNOWN} explicitly rather than hiding it: BTC-USDT has 13.19\% overall \texttt{UNKNOWN} share with daily p90 15.42\%, and ETH-USDT has 12.68\% overall \texttt{UNKNOWN} share with daily p90 15.59\%. Sensitivity is evaluated on deterministic one-million-row samples per asset. Under stricter extreme quantiles, \texttt{UNKNOWN} rises to 16.27\% for BTC and 15.78\% for ETH, with agreement to the baseline taxonomy of 86.48\% and 85.75\%, respectively. Under relaxed extremes, \texttt{UNKNOWN} falls to 10.57\% and 10.18\%, with agreement of 86.85\% and 86.31\%. These sensitivity results are diagnostic; they are not used to retune thresholds on test data.

By-regime evaluation is necessary because a model can look acceptable under an aggregate macro-F1 or MCC while concentrating errors, cost, or turnover in unfavorable states. Execution metrics are even more regime-sensitive because spread, available depth, and latency penalties are not constant across market conditions. A signal that is marginal in a calm liquid book may be dominated by adverse selection or cost in a momentum-toxic or liquidity-drought state. Thus, the taxonomy provides a bridge from prediction metrics to selective-execution diagnostics: if a policy reduces exposure in some regimes but fails in others, the mixed result is treated as evidence rather than hidden.

\subsection{Forecasting Baselines}
We use tabular and temporal baseline families: an SGD-trained linear classifier, a GPU XGBoost classifier \cite{chen2016xgboost}, temporal convolutional networks (TCNs) \cite{bai2018tcn}, and implementation-validated DeepLOB-style CNN--LSTM and lightweight LOB-Transformer baselines inspired by the LOBFrame model suite \cite{briola2025lobframe}. All temporal deep baselines use the same causal tensor contract, namely 100-event windows over the first 10 LOB levels, represented as 40 relative-price and log-size features. The locked full-row execution claims in this paper are made only for models with generated prediction and execution artifacts. DeepLOB-style and Transformer-style models are therefore treated as canonical deep LOB coverage in the artifact suite and as pilot/smoke baselines unless their full-row artifacts are explicitly reported.

The baselines are diagnostic controls rather than a claim of state-of-the-art architecture development. Table~\ref{tab:forecasting} shows that rankings depend on the metric. XGBoost has the highest accuracy, SGD and XGBoost have similar MCC, while the stride-1 TCN has the strongest macro-F1 among full-row comparable baselines but lower MCC. This disagreement motivates execution-aware evaluation.



Forecasting performance is reported with four metrics because the cost-aware ternary task is class-imbalanced. Accuracy rewards the majority class, macro-F1 gives equal weight to UP, FLAT, and DOWN, MCC provides a conservative correlation-style summary, and balanced accuracy averages recall across classes. None of these metrics is treated as a deployment criterion. They are used to diagnose whether predictive structure exists before the same predictions are passed through execution.

\section{Forecast-to-Execution Evaluation}
\subsection{L2 Snapshot Replay Specification}
The execution layer converts predicted actions into market-order-style replay against visible L2 depth. This replay is a benchmark approximation: it does not model passive queue priority, hidden liquidity, cancellation races, venue routing, or live latency. Spread crossing is implicit because buys execute on ask levels and sells execute on bid levels.

For a nonzero signal at row $i$, entry occurs at $i+\ell$ where $\ell$ is the configured latency in event rows, and exit occurs after the label horizon. The requested quantity at a replay snapshot is
\begin{equation}
q_t^*=\frac{N}{\max(m_t,\epsilon)},
\end{equation}
where $N$ is the configured trade notional, $m_t$ is the snapshot mid-price, and $\epsilon=10^{-9}$. A market buy sweeps asks from level $0$ to level $19$; a market sell sweeps bids in the same level order. Under spread and depth stress, the executable prices and visible sizes are
\begin{align}
p^{\rm buy}_{t,l}&=m_t+\alpha(p^{\rm ask}_{t,l}-m_t), &
d^{\rm buy}_{t,l}&=\beta d^{\rm ask}_{t,l},\\
p^{\rm sell}_{t,l}&=m_t-\alpha(m_t-p^{\rm bid}_{t,l}), &
d^{\rm sell}_{t,l}&=\beta d^{\rm bid}_{t,l},
\end{align}
where $\alpha$ is the spread multiplier and $\beta$ is the depth multiplier. At each level, the fill is $x_l=\min(q_{\rm remaining},\max(d_{t,l},0))$. Nonfinite, nonpositive, or missing deeper-level price/size values are treated as unavailable depth for that level.

The execution price is the visible-depth VWAP,
\begin{equation}
\mathrm{VWAP}_t=\frac{\sum_l x_l p_{t,l}}{\sum_l x_l},
\end{equation}
with fill ratio $\sum_l x_l/q_t^*$. If \texttt{partial\_fill=true}, a trade is kept only when both entry and exit fill ratios are at least the configured \texttt{min\_fill\_ratio}; if \texttt{partial\_fill=false}, any non-full sweep is converted to a zero fill and skipped. The matched trade quantity is the smaller of entry and exit filled quantities.

Fees are charged on matched filled notional, not requested notional:
\begin{equation}
C=\frac{f}{10000}\left(q p_{\rm entry}+q p_{\rm exit}\right),
\end{equation}
where $f$ is the fee in basis points. Long gross PnL is $q(p_{\rm exit}^{\rm sell}-p_{\rm entry}^{\rm buy})$; short gross PnL is $q(p_{\rm entry}^{\rm sell}-p_{\rm exit}^{\rm buy})$; net PnL subtracts $C$. Missing required replay columns fail fast. Rows with nonfinite or nonpositive mid-price, invalid best prices or sizes, or crossed best books are treated as zero-fill rows. These rules make depth exhaustion, fee application, latency indexing, and partial fills reproducible from L2 snapshots.

The execution layer evaluates three decision mappings. The naive threshold policy trades when the predicted UP or DOWN probability exceeds a validation-tuned threshold. The cost-aware threshold policy estimates whether expected directional return exceeds spread, fee, and depth-slippage proxies. RSEP extends this gate with explicit latency, liquidity, adverse-selection, and regime-risk penalties. The RSEP $\lambda$ coefficients are fixed benchmark constants from a singleton configuration grid, while $\theta$ is selected on validation data. The held-out test split is used only for final reporting and is not used to choose favorable execution assumptions.

\subsection{RSEP Gate}
RSEP gates a forecast-derived action by comparing forecast-implied expected edge with explicit cost and risk terms. Let $\mu_y$ denote the training-split mean future return for class $y\in\{\textsc{Up},\textsc{Flat},\textsc{Down}\}$, and let $p_t(y)$ be the forecast probability at time $t$. The implementation uses
\begin{equation}
\widehat{e}_t=
p_t(\textsc{Up})\mu_{\textsc{Up}}+
p_t(\textsc{Flat})\mu_{\textsc{Flat}}+
p_t(\textsc{Down})\mu_{\textsc{Down}}.
\label{eq:rsep_edge}
\end{equation}
The cost term is $\widehat{c}_t=\rho_t+f/10000$, where $\rho_t$ is relative spread and $f$ is the simulator fee in basis points. The gate trades only if
\begin{equation}
\widehat{e}_t>
\widehat{c}_t+
\lambda_{\rm lat}\widehat{r}^{\rm lat}_t+
\lambda_{\rm liq}\widehat{r}^{\rm liq}_t+
\lambda_{\rm adv}\widehat{r}^{\rm adv}_t+
\lambda_{\rm reg}\widehat{r}^{\rm reg}_t+\theta.
\label{eq:rsep}
\end{equation}
For negative edge, the symmetric short rule is used: sell if $\widehat{e}_t<-\widehat{E}^{\rm req}_t$, where $\widehat{E}^{\rm req}_t$ is the right-hand side of Eq.~\eqref{eq:rsep}. Otherwise the policy abstains. Equality abstains because both comparisons are strict. RSEP is not a forecasting model and not a universal trading policy. It is an interpretable diagnostic for testing whether selective execution reduces exposure when estimated edge is too small relative to cost, latency risk, liquidity risk, adverse-selection risk, and regime risk.

\section{Stress, Robustness, and Failure Analysis}
\subsection{Stress Grid}
We stress the execution layer along four axes with a fixed benchmark grid: fee $f\in\{0,1,2,5,10\}$ basis points, latency $\ell\in\{0,1,5,10\}$ snapshot events, spread multiplier $\alpha_s\in\{1.0,1.5,2.0\}$, and visible-depth multiplier $\alpha_d\in\{1.0,0.75,0.5\}$. The default replay setting is $f=1$ bps, $\ell=1$ event, $\alpha_s=1.0$, and $\alpha_d=1.0$. We perturb exactly one axis at a time; all other execution assumptions remain at their default values. Stress results are diagnostic perturbations of the L2 snapshot replay approximation, not claims about live execution.


The stress grid is intentionally applied after model training and validation tuning. The held-out predictions, train-derived class-return means, selected policy thresholds, RSEP $\lambda$ constants, and validation-selected RSEP $\theta$ are fixed before stress replay. We do not retrain models, recompute probabilities, or retune thresholds under any stress level. This makes the stress results closer to an out-of-distribution execution diagnostic: the forecast and decision parameters are fixed, while the execution environment changes. A robust model-policy mapping should degrade less severely when explicit cost, timing, or liquidity assumptions become less favorable.

\section{Evidence Mapping and Reproducibility}
The benchmark is organized so that each major paper claim has a direct artifact path. This is important because the empirical narrative contains both positive and negative evidence. Positive evidence includes the existence of forecasting signal, the ability to report degradation by regime, and relative loss mitigation under RSEP in selected settings. Negative evidence includes net-negative execution, stress sensitivity, and the stride-1 TCN result where stronger macro-F1 does not translate into better RSEP behavior. Without explicit evidence mapping, a reader could mistakenly interpret the paper as a trading-profit claim or, conversely, treat negative net PnL as a failed experiment. The artifact discipline prevents both errors.

The reproducibility protocol therefore separates three layers. The first layer is data construction: audit statistics, cleaning decisions, feature/label stores, and chronological split metadata. The second layer is modeling: forecasting predictions, model labels, validation-tuned thresholds, and test-only reporting. The third layer is evaluation: execution replay outputs, stress-grid results, by-regime diagnostics, ablation tables, and bootstrap confidence intervals. Each layer is generated before the paper narrative is finalized, so the draft does not introduce numbers that cannot be traced to a canonical table or figure.

Where the results are mixed, the text says so explicitly. Finally, the benchmark records enough information for independent reruns under the same configuration: data version, symbol, date range, feature version, split definition, model label, validation threshold, execution cost configuration, stress axis, bootstrap unit, and output artifact path. This is the appropriate level of reproducibility for a large-scale financial benchmark built from licensed L2 data.

The artifact package therefore separates restricted data from public reproducibility assets. The released files include schema documentation, benchmark configs, checksum manifests, paper-ready tables, split audits, and a synthetic L2 sample with the same 20-level snapshot schema. A one-command smoke pipeline runs on this synthetic sample and exercises data audit, feature construction, cost-aware labels, regime assignment, chronological splitting, forecasting, execution replay, RSEP, stress-grid evaluation, and report generation. The synthetic sample is not used for scientific claims; it only verifies that the released code paths are executable without commercial raw data.


\section{Threats to Validity}
\subsection{Internal Validity}
The main internal-validity concern is leakage. LOB data are highly autocorrelated, and even small violations of temporal order can inflate both forecasting and execution metrics. The benchmark mitigates this risk by using chronological splits, training-split normalization, validation-only policy tuning, and test-only reporting. Future mid-price values are used only to define labels. They are not used in features, regime assignments, threshold fitting, or execution decisions. The regime taxonomy is likewise causal: rolling diagnostics end at the decision time, and quantile thresholds are fitted before test evaluation. These safeguards do not prove that no implementation bug exists, but they define the intended leakage boundary and make it auditable.

A second internal-validity concern is policy overfitting. Because RSEP has several risk weights, it could be over-tuned if thresholds were selected from test results. The paper prevents this by treating validation as the only tuning split and by reporting negative cases. The stride-1 TCN is the most important check here: if the paper had tuned RSEP only to support a favorable narrative, the TCN evidence would likely be omitted. Keeping this result in the main text shows that the benchmark is not optimized to make RSEP appear universally superior.

We also maintain a model-selection ledger for the reported Stage 3 experiments. The ledger records the forecasting model family, feature and label versions, prediction artifact, threshold grid, RSEP grid, validation objective, selected thresholds, validation result, and held-out test result for each model-policy scope. Within-asset execution thresholds are selected only on chronological validation rows; asset-held-out execution thresholds are selected only on source-asset validation rows; and stress-grid results keep predictions and selected thresholds fixed. The paper does not claim that data-snooping risk is eliminated. A formal Reality Check or PBO/CSCV diagnostic would require per-candidate daily returns for every tried configuration, whereas the current saved artifacts retain full daily/trade returns only for selected policies. We therefore report conservative controls--the full candidate ledger, validation-only selection, test-only reporting, day-level bootstrap, no retuning under stress, and retained negative evidence--rather than presenting an unsupported overfitting $p$-value.

\subsection{Construct Validity}
Construct validity concerns whether the benchmark measures the intended concept. The target concept is not live trading profit; it is forecast-to-execution degradation under observable L2 frictions. The L2 replay captures spread crossing, visible-depth sweep, explicit fee, latency shift, and stress perturbations. It does not capture hidden liquidity, queue position, order cancellations, market impact beyond the visible-depth approximation, or venue-specific routing. Therefore, net PnL in this paper should be read as a diagnostic quantity produced by a controlled replay approximation. It is useful for comparing degradation under the same assumptions, but it is not an estimate of live deployable profit.

The cost-aware label also affects construct validity. If the threshold is too small, the task collapses toward ordinary direction prediction and may overstate actionable signals. If it is too large, many observations become FLAT and the task may understate weak but real predictive structure. The chosen threshold is therefore part of the benchmark definition. The paper reports forecasting metrics, execution results, and stress tests together so that readers can see how this modeling choice propagates into downstream decisions.

\subsection{External Validity}
External validity is intentionally narrow. The benchmark evaluates Binance BTC-USDT and ETH-USDT in 2024. The results should not be generalized to all crypto assets, all venues, equities, futures, or foreign exchange without new experiments. Cross-asset BTC$\leftrightarrow$ETH results strengthen the benchmark relative to a single within-asset split, but they remain two directions inside a related pair of highly liquid crypto markets. Assets with lower liquidity, different fee tiers, different market-making behavior, or different exchange microstructure may produce different degradation profiles.

External validity is also limited by the snapshot frequency and the available depth levels. A higher-frequency message feed, deeper book, or market-by-order dataset could change estimated slippage, latency sensitivity, and passive-fill conclusions. This does not invalidate the benchmark. It means that CryptoRegimeShift-LOB should be viewed as a reproducible L2 diagnostic layer. It can be extended when richer data become available, but the present claims remain limited to the L2 snapshot setting.

\subsection{Interpretive Validity}
The final threat is interpretive: readers may be tempted to treat lower losses as success or negative net PnL as failure. Both interpretations are incomplete. Lower losses under RSEP show that selective gates can filter some adverse conditions, but they do not prove profitability. Negative net PnL shows that the forecasting signal is thin after cost and stress, but it does not make the benchmark uninformative. The paper's contribution is precisely to expose this gap. A prediction-only benchmark might stop after reporting macro-F1; CryptoRegimeShift-LOB continues until the signal is mapped through execution, stress, and cross-asset diagnostics.


\section{Discussion}
\subsection{Practical Benchmark Takeaways}
The empirical pattern across sections is consistent: forecasting quality is real but fragile. The models detect short-horizon structure in the L2 book, yet the signal changes character after the decision is exposed to spread crossing, fee, latency, depth, and regime stress. The practical value of the benchmark is not a single trading score. It is an audit trail from prediction to degradation. A practitioner can use the same path to ask where a model fails: before costs, after spread crossing, under latency, in a liquidity regime, or under BTC$\leftrightarrow$ETH transfer. This makes negative net PnL informative because it identifies which part of the financial ML pipeline destroys the apparent signal.

The first takeaway is that model ranking is evaluation-dependent. XGBoost has the highest accuracy, stride-1 TCN has the strongest macro-F1 among full-row comparable baselines, and the tabular models remain more competitive under MCC and execution diagnostics. No single metric orders the models consistently. This disagreement is not noise to be averaged away; it is evidence that LOB forecasting quality is multidimensional. CryptoRegimeShift-LOB lets users check whether a model that improves one prediction metric remains useful after turnover, cost, and selective-execution gates are considered.

The second takeaway is that the stress grid localizes failure modes. Fee stress dominates the reported grid, but latency, spread, and depth remain necessary axes because their importance can change under different venue assumptions or trade sizes. Reporting stress endpoints and robustness areas prevents the evaluation from depending on one favorable cost configuration. In applied financial settings, such sensitivity is often more informative than a single backtest score because it tells users which execution assumption breaks the signal first.

The third takeaway is that selective execution can be compared reproducibly. RSEP, naive thresholds, and cost-aware thresholds are evaluated with fixed predictions, validation-only tuning, and day-level bootstrap. This makes the comparison useful even when all policies remain net negative: the benchmark measures whether a gate reduces exposure, shifts regime concentration, or fails under a stronger forecasting model. The stride-1 TCN result is important here because it prevents a simplistic narrative in which better macro-F1 automatically implies better selective execution.

Finally, cross-asset evaluation changes the evidential standard. Within-asset tests show that the pipeline works on one chronological split; BTC$\leftrightarrow$ETH tests ask whether the same source-tuned pipeline produces interpretable degradation on a related but different asset. Reporting the two directions separately exposes asymmetry in price scale, relative spread, depth, label mix, regime distribution, and calibration. This gives future users a reusable template for adding a new asset or venue: rerun the same evidence path, report the same diagnostics, and upgrade claims only when the new observational layer supports them.

\section{Limitations and Responsible Use}
CryptoRegimeShift-LOB is bounded by snapshot-level L2 data. This boundary is useful for a public benchmark because visible-depth replay can be specified, tested, stressed, and reused across models. It also defines what the benchmark cannot observe: exact order arrivals, cancellations, queue priority, hidden liquidity, passive-fill dynamics, venue routing, and matching-engine details. Live deployment would require additional venue-specific evidence, including order-routing logs, measured latency, market-impact analysis, operational risk controls, and compliance review.


The forecasting baselines are a controlled diagnostic suite rather than a SOTA model-development claim. The execution results remain net negative across main reported policies, which is a core empirical finding of the benchmark: gross directional signal can be overwhelmed by spread crossing, fees, latency, visible-depth constraints, and regime-dependent risk. RSEP is useful as a diagnostic probe because it shows when selective gates reduce exposure or losses and when they fail; the stride-1 TCN is explicit negative evidence because cost-aware execution is better than RSEP and the RSEP-vs-cost-aware bootstrap interval crosses zero.

The regime taxonomy is an evaluation device rather than a true latent-state model. The cross-asset analysis is limited to BTC-USDT$\leftrightarrow$ETH-USDT and should be used as directional transfer evidence, not as a universal cross-market result. The benchmark is intended for research evaluation, robustness analysis, and risk-aware model assessment. It should not be interpreted as investment advice, trading recommendation, or evidence that an automated strategy is ready for deployment. This responsible-use boundary is especially important in crypto markets, where market integrity, liquidity fragmentation, operational risk, and retail exposure remain policy concerns \cite{iosco_crypto2023,fsb2023crypto}.


Release and reuse should therefore emphasize reproducibility and risk assessment. The benchmark can support comparisons of forecasting models, cost-aware labels, stress-grid assumptions, selective-execution gates, and asset-transfer diagnostics. Future extensions can add L3 data, passive order simulation, venue-specific fee schedules, or additional assets by reusing the same evidence map and adding the missing observational layer before strengthening the claim.

The checklist is not a substitute for public raw data access, but it reduces ambiguity in the paper. For a licensed financial dataset, raw snapshots may be too large or contractually restricted. The reproducibility burden therefore shifts to code paths, configuration files, split definitions, model identifiers, and paper-ready artifact tables. This is why the manuscript avoids unverifiable one-off numbers. If a statistic is not locked by a table, figure, or audit output, it is either omitted or described qualitatively.

This structure also helps future extensions. A researcher who adds L3 data, passive order simulation, another venue, or a new model family can reuse the same evidence map. The new experiment would need to add the missing observational layer rather than silently upgrading the claim. For example, a future L3 extension could claim queue-position analysis only after adding message-level order events and corresponding simulator tests. The current paper does not make that claim, and the checklist makes the limitation explicit.

\section{Conclusion}
This paper introduced CryptoRegimeShift-LOB, a regime-aware benchmark and evaluation protocol for snapshot-level L2 crypto order book forecasting. The benchmark connects causal features, cost-aware ternary labels, interpretable regimes, forecasting baselines, L2 snapshot execution replay, stress grids, bootstrap diagnostics, and BTC$\leftrightarrow$ETH asset-held-out evaluation. The empirical evidence shows that forecasting signal and execution usefulness can diverge: a model may improve directional metrics while remaining fragile after costs, stress, regime shifts, or asset transfer. CryptoRegimeShift-LOB provides an applied evaluation path for measuring that divergence reproducibly. For high-frequency financial decision contexts, credible LOB evaluation should be cost-aware, regime-aware, stress-tested, and explicit about the observational limits of the data.
